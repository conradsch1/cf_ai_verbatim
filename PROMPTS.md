### Feature/Task: Debug — practice stuck after Step 2 (UI not advancing)
* **Goal:** Fix inline practice so that after a correct check on Step 2 (or 3), the UI advances instead of staying on the same step; verify with runtime evidence and document the investigation.
* **My Prompt (original issue):** "With backend + frontend dev running, on Step 2 or 3, after typing the last word correctly: 'Checking…' shows then goes away, but the UI does not advance to the next step and does not reset — nothing visible changes."
* **Debugging:** Debug-mode workflow with NDJSON logs (ingest + `.cursor/debug-2a9b95.log`). Instrumented [`frontend/src/PracticeInline.tsx`](frontend/src/PracticeInline.tsx) (`runCheck`, `handleKeyDown`) and [`frontend/src/App.tsx`](frontend/src/App.tsx) (`handlePracticeUpdate`). Logs showed the client on `step: 2` while `POST /check` returned `correct: true` but `practice.step` still **2** (expected **3**) — i.e. the server was advancing **1 → 2** again, not **2 → 3**. Ruled out frontend dropping updates; confirmed server-side. **Root cause:** `MemorizationSession` held practice cursor only in memory; Durable Object eviction between HTTP requests reset state, so the next check ran against a fresh step-1 cursor while the UI still reflected the previous response. **Fix:** persist and restore `practice` (and `sm2`) via `ctx.storage` in [`backend/src/memorization-session.ts`](backend/src/memorization-session.ts) (`blockConcurrencyWhile` load in constructor; `put` after `/practice/sync`, `/practice/advance`, `/practice/retry`). Related hardening already in tree: [`backend/src/practice-api.ts`](backend/src/practice-api.ts) builds the post-check payload from `advanceDO` state; `advanceAfterSuccess` uses `Number(p.step)` for branch matching. Post-fix logs: Step 2 check → `practiceStep: 3` and parent `step: 3`. Removed temporary ingest `fetch` logs from the frontend after verification.
